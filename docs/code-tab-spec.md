# Code tab — спецификация и состояние реализации

> Статус на 2026-07-15: стадии 0–6 и Workspace 1.0.52 реализованы. Версия 1.0.50 добавила
> типизированное рабочее дерево `A/M/D/R`, а 1.0.52 — постоянные вкладки, Quick Open, поиск по
> репозиторию и историю переходов назад/вперёд.

## Концепт

Нижний таб **Code** = мини-IDE внутри репозитория:
**переосмысленный браузер «рабочего дерева» → существующий редактор (`CodeEditorScreen`) → батч-коммит.**

Это отдельная секция, не равная read-only лендингу репозитория (файловому дереву на входе).
`selectedSection = Code` при заходе в таб.

## Архитектура

- Редактор остаётся **«тупым»**: на вход `content + path + language`, на сохранение — **колбэк save-to-draft**. Редактор НЕ знает про GitHub.
- Всю GitHub-логику (фетч/коммит) владеет **таб Code**, не редактор.
- Внутренняя навигация Code-таба: `browser → editor → back в browser` (свой бэк-стек, не выход из репо).
- **Стрелка назад из корня браузера → выход из Code на лендинг Repo.**
- Детект языка для редактора — по расширению файла.

## Данные: черновик + батч-коммит

### Черновик (draft)
- Незакоммиченные правки = **черновик**, скоуп **`(репо, ветка)`**.
- Персистится **на диск** (переживает смерть процесса).
- **Авто-сохраняется молча** на: выходе из Code / смене ветки / уходе в фон. Без диалогов.
- Авто-восстанавливается при возврате в Code: грязные маркеры на месте, панель «изменения» показывает «N несохранённых».
- Отдельного UI у черновика нет — он виден через панель «изменения», там же и discard.
- Буфер: `path → CodeChange` (`CodeAdd`, `CodeModify`, `CodeDelete`, `CodeRename`).
- Старый дисковый формат `path → content` автоматически мигрирует в `CodeModify`.

### Смена ветки
- Авто-дрфт текущей ветки + загрузка черновика целевой ветки (если есть).
- Дилемма «оставить/сбросить» растворяется — каждая ветка держит свой черновик. Discard остаётся опцией.

### Коммит (Git Data API, батчем)
Один коммит на много файлов:
1. Получить ref (текущая ветка) → sha последнего коммита.
2. Получить tree sha этого коммита (base tree).
3. Для каждого изменённого файла: создать blob → sha.
4. Создать tree (`base_tree` + записи: модификации/добавления через blob sha, удаления через `sha:null`, переименование = удаление старого + добавление нового).
5. Создать commit (message, tree, parent = последний коммит).
6. Обновить ref → новый sha коммита.

Коммитим в **текущую выбранную ветку** (селектор сверху).

### Конфликт (ref уехал во время коммита)
Перефетч ref, предупредить пользователя, дать ретрай. **Без авто-мёрджа.**

## Переосмысленный браузер (changes-oriented)

Отличие от лендинга: лендинг = «читаю репо», Code-браузер = «рабочее дерево с правками».

- **Панель «изменения»**: файлы с незакоммиченными правками + кнопка **«Закоммитить N файлов»** → шит сообщения → батч. (По сути лёгкий Source Control как в VS Code.)
- **Маркеры грязных**: точка у изменённого файла; роллап-точка на папке, где внутри есть правки.
- **«Недавние / открытые»** строкой сверху — быстрый возврат к файлу.
- **Крошки пути + быстрый «вверх»**.
- **Create / rename / move / duplicate / delete** работают как локальные операции черновика.
- Пустая папка представляется файлом `.gitkeep`, потому что Git не хранит пустые директории.
- Фильтр файлов.

## Зафиксированные решения

| # | Решение |
|---|---------|
| 1 | Назад из корня браузера → выход из Code на лендинг Repo. |
| 2 | Смена ветки с правками → авто-черновик per-ветка (без блокировки). |
| 3 | Долговечность правок → черновик персистится на диск. |
| 4 | Create/rename/move/duplicate/delete → реализовано в 1.0.50 как операции черновика. |
| 5 | С 1.0.52 открытые вкладки сохраняются отдельно для каждой ветки. |
| 6 | Конфликт ref → перефетч + предупреждение + ретрай, без авто-мёрджа. |

## Стадии сборки

Дисциплина: **build + commit + device-test на каждой стадии. Без чейнинга.**

| Стадия | Содержание |
|--------|------------|
| 0 | Оболочка Code + внутренний бэк-стек + проводка `selectedSection` (заглушка-экран). |
| 1 | Переосмысленный браузер **read-only** — фетч дерева, крошки, навигация, иконки по типу. |
| 2 | Открыть файл → существующий редактор **read-only** — фетч контента, детект языка, показ. *(Тут же ложится отложенный Phase-2 state-lift редактора — оборачиваем, НЕ переписываем.)* |
| 3 | Правка → **буфер черновика** + маркеры грязных + discard. Ничего не пушится. |
| 4 | **Батч-коммит** via Git Data + шит сообщения. **Единственная реальная рисковая часть (дата-слой), изолирована тут.** |
| 5 | Панель «изменения» + полиш — отдельная поверхность изменений, роллап-маркеры, недавние/открытые. |
| 6 | Прочность — гард смены ветки, персист черновика на диск, обработка конфликтов. |

Стадии 0–6 завершены. Батч-коммит поддерживает add/modify/delete/rename одним tree и обновляет ref
с `force=false`; при гонке ветки UI сохраняет черновик и предлагает retry.

## Workspace 1.0.52

- Открытые вкладки, активный файл и история переходов хранятся на диске в скоупе
  `(репозиторий, ветка)`; после перезапуска возвращается то же рабочее пространство.
- Закрытие вкладки не удаляет draft. Переключение вкладки, Quick Open и переход по истории сначала
  передают последние правки в типизированный draft.
- Quick Open строит индекс по рекурсивному Git tree выбранной ветки, накладывает локальные
  `A/M/D/R` и исключает бинарные файлы из текстового редактора.
- Search repo использует GitHub Code Search для индексированного содержимого и объединяет результат
  с поиском по незакоммиченным локальным draft-файлам. GitHub индекс ищет default branch; это явно
  отмечается в интерфейсе, если workspace открыт на другой ветке.
- Back/forward история ограничена 40 переходами, вкладки — 12 файлами, чтобы состояние оставалось
  компактным и предсказуемым на телефоне.
