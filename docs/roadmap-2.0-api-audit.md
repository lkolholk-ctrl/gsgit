# Роадмап 2.0.0 — полный аудит GitHub REST API

Цель: пройтись по каждой дата-модели и каждому домену `data/github/`, сверить с
официальным REST API GitHub, добить недостающее, починить неработающее.
Это второе большое обновление после публичного релиза 1.x.

## Метод (не глазами, а по спеке)

Источник истины — официальная OpenAPI-спецификация GitHub:
`github/rest-api-description` (файл `descriptions/api.github.com/api.github.com.json`).

На каждый домен:
1. Выгрузить из спеки схемы ответов соответствующих эндпоинтов.
2. Механически сравнить с нашей моделью (`model/GHX.kt`) и `parse*`-функциями:
   - недостающие поля (есть в API, нет у нас);
   - мёртвые поля (есть у нас, нет в API);
   - расхождения типов/nullability (крэши на `!!`/`getString`);
3. Сверить покрытие эндпоинтов домена: что из спеки вообще не реализовано.
4. Отметить deprecated/sunset эндпоинты (прецедент: classic Projects → 410).
5. Проверить требуемые permissions GsGit App для каждого эндпоинта.
6. Ручной прогон экрана, который домен обслуживает (по audit-playbook.md).

## Порядок доменов (по важности для юзера)

1. Repos + Contents (файлы/браузер — ядро)
2. Issues + Comments + Labels + Milestones
3. PullRequests + Reviews + Checks (merge-флоу целиком)
4. Actions + Workflows + Artifacts (+ наши воркеры загрузок)
5. Commits + Branches + Tags + Compare
6. Releases + Assets
7. Users + Orgs + Teams
8. Notifications
9. Search
10. Discussions
11. Gists
12. Webhooks
13. ProjectsV2 (GraphQL — сверка по GraphQL-схеме)
14. Secrets / Deploy keys
15. Stars/Forks/Watching + Reactions
16. Настройки репозитория (GitHubRepoSettingsManager)

## Правила процесса

- Один домен = одна сессия (`/clear` между ними), как в audit-playbook.md.
- Сначала отчёт расхождений, правки — только после подтверждения.
- Каждый домен: отчёт → фикс → билд зелёный → ручная проверка экрана → следующий.
- Definition of Done домена: покрытие эндпоинтов осознанное (реализовано или
  явно решено «не нужно»), ноль небезопасного парсинга, permissions сверены,
  экран прогнан руками.

## Кандидаты в 2.0.0 сверх аудита (копятся здесь)

- **Мульти-аккаунт.** Несколько GitHub-аккаунтов в приложении с быстрым
  переключением — для удобства. Решения зафиксированы:
  - **Лимит — 3 аккаунта** на первое время.
  - **Вход независимо на каждый** — device flow ИЛИ токен, по выбору (переиспользуем
    экран входа с двумя дверями + `authMode` на аккаунт).
  - **Хранение** — список сессий per-account в `TokenRepository` + активный аккаунт;
    у каждого свой токен (device/pat) и `authMode`.
  - **Переключение — как в Gmail:** тап по аватару → список аккаунтов; «добавить
    аккаунт» ведёт на экран входа.
  - **Пуши — только на АКТИВНЫЙ аккаунт.** При переключении устройство
    перерегистрируется на новый логин (старый — снять). Серверных правок минимум:
    используем существующие register/unregister, схема «один FCM-токен = один
    активный логин» — БЕЗ мульти-логина на устройство.
  - **PIN/защита приложения — общая** на всё приложение (не на аккаунт).
  - **Раздельный кэш пользователя и ETag на аккаунт** (сброс/своп при переключении).
- Реалтайм джобы/логи Actions через сервер (отложено осознанно, см. обсуждение).
- Обновить «60k+ lines of kotlin» на лендинге (уже 75k+).
